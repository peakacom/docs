---
title: "How to mask sensitive columns with tags"
description: "Learn how to protect PII and other sensitive data in Peaka by tagging columns and applying masking rules per user or group."
---

Column masking lets you hide or transform sensitive values — emails, phone numbers, national IDs, salaries — without changing the underlying data or writing a separate query for every audience. You classify columns with **tags**, and each tag carries **mask rules** that decide what each user or group sees. Masking is enforced at query time by Peaka's query engine, so it applies everywhere the data is consumed: the Peaka Studio, the Query API, BI tools, and embedded apps.

## How it works

Three building blocks work together:

- **Tags** classify your data (for example `PII`, `Financial`, `Confidential`). A tag can be attached to any number of columns across your catalogs.
- **Masks** define *how* a value is transformed. You can use Peaka's predefined masks or create your own with a SQL expression.
- **Mask rules** connect the two: for a given tag, a rule says *who* (a specific user, a group, or everyone) sees the data through *which* mask.

When a query touches a tagged column, Peaka picks the most specific rule that matches the querying user and rewrites the column on the fly:

1. A rule targeting the **user** directly wins first,
2. then a rule targeting one of the user's **groups**,
3. then a rule targeting **everyone**.

If no rule matches — or the tag is inactive — the column is returned unmasked.

## 1. Open Data Governance settings

Tags and masks are managed at the organization level. In Peaka Studio, open your organization settings and navigate to the **Data Governance** section. You need the *Manage Data Governance* permission to see and edit these settings.

## 2. Pick a mask (or create a custom one)

Peaka ships with predefined masks that cover the most common cases:

| Mask                         | What it does                                                        | Example (`jane.doe99@acme.com`) |
| ---------------------------- | ------------------------------------------------------------------- | -------------------------------- |
| Redact                       | Replaces letters with `X`/`x` and digits with `0`                    | `xxxx.xxx00@xxxx.xxx`            |
| Partial Mask Show Last Four  | Masks everything except the last four characters                     | `XXXXXXXXXXXXXXX.com`            |
| Partial Mask Show First Four | Masks everything except the first four characters                    | `janeXXXXXXXXXXXXXXX`            |
| Nullify                      | Replaces the value with `NULL`                                       | `NULL`                           |
| Unmasked                     | No masking — useful for exempting a user or group from a tag's rules | `jane.doe99@acme.com`            |
| Date Show Only Year          | Truncates a date to its year                                         | `2024-01-01` (from `2024-08-15`) |

If none of these fit, create a **custom mask** with a SQL expression. Use the `{col}` placeholder for the column and `{type}` for its data type — Peaka substitutes them at query time. For example, a mask that keeps only the email domain:

```sql
cast(concat('***@', split_part({col}, '@', 2)) as {type})
```

## 3. Create a tag with mask rules

Create a tag (for example `PII`) and add its mask rules. A typical setup:

- **Everyone** → *Redact* — by default, nobody sees raw values.
- **Group: Data Team** → *Unmasked* — analysts who need the raw data are exempted.
- **User: support@acme.com** → *Partial Mask Show Last Four* — support can verify identities without seeing full values.

Keep the tag **active** — deactivating a tag switches off all of its masking at once, which is useful for testing.

## 4. Assign the tag to columns

Open the table that contains sensitive data and assign the tag to the relevant columns — for example the `email` and `phone` columns of your CRM catalog. A column can carry one tag; the same tag can be reused across any number of tables and catalogs.

## 5. Verify the masking

Run a query on the tagged table as different users:

```sql
SELECT id, full_name, email, phone FROM crm.public.customers LIMIT 5
```

A member of the *Data Team* group sees raw values, while everyone else sees the masked version:

| id  | full_name    | email                 | phone          |
| --- | ------------ | --------------------- | -------------- |
| 1   | Xxxx Xxx     | xxxx.xxx00@xxxx.xxx   | 000-000-0000   |

## Things to keep in mind

- Masking is applied by the query engine, so it also covers downstream access paths — the Query API, BI tool connections, and embedded Peaka.
- Rule precedence is *user → group → everyone*; only the single most specific rule is applied.
- Custom mask expressions run inside the query, so any Peaka SQL function is available in them.
- Masks transform values but do not remove rows. To control *which rows* a user can see, combine masking with [row-level security](/how-to-guides/how-to-implement-row-level-security).
